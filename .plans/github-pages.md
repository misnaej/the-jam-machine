# Plan: GitHub Pages Documentation Site

**Date:** 2026-03-27 (revised)
**Status:** Planned

---

## Goal

Create a documentation site on GitHub Pages explaining how The Jam Machine works. Target audience: developers and musicians who want to understand the MIDI-to-text encoding, the model, and the generation pipeline.

---

## Revised Scope — 3 PRs

The previous plan tried to do everything at once. This revision splits into independent PRs that each deliver value.

### PR 1: Minimal site — Landing page + Encoding guide (MVP)

Set up GitHub Pages with Jekyll (markdown → HTML) and write the two core pages.

**Files:**
- `docs/index.md` — Landing page
- `docs/encoding-decoding.md` — Encoding/decoding guide with Reptilia examples
- `docs/_config.yml` — Jekyll theme config
- `docs/assets/` — Pre-generated images (piano roll, pipeline diagrams)
- `README.md` — Link to the site

**Content for `index.md`:**
- What is The Jam Machine (1 paragraph)
- How it works (the pipeline diagram from README, refined)
- Token vocabulary table
- Links: live demo, encoding guide, GitHub repo

**Content for `encoding-decoding.md`:**
- From MIDI to Text (with code snippets from `midi_codec.py`)
- Token format and vocabulary
- Quantization caveats (from `midi_codec.py` docstring)
- From Text back to MIDI
- Worked example: Reptilia snippet (pre-generated, committed as assets)

**Why Jekyll?** GitHub Pages renders it automatically — no build step, no CI needed. Write markdown, push, done. Can upgrade later.

**Pre-generate assets before writing docs:**
```bash
pipenv run python examples/encode_decode.py  # generates piano roll
pipenv run python examples/track_stats.py     # generates split plots
```
Commit the PNGs to `docs/assets/`.

### PR 2: Track Analysis page (optional, after MVP)

Extend `track_stats.py` to generate an HTML report with embedded visualizations. This is the "case study" page.

**Visualizations (prioritized):**
1. Piano roll (already exists)
2. Per-instrument timeline with coverage stats
3. Split instrument plots (already exist)
4. Token frequency chart
5. Encoding comparison table (original vs decoded)

Skip the less useful ones (density heatmap, pitch distribution, quantization impact) unless they add clear educational value.

**Format:** Self-contained HTML with base64-embedded PNGs. Generated by `scripts/build-track-analysis.py`.

### PR 3: Notebook cleanup + Embedding explorer page

Fix the `exploring_the_embedding.ipynb` notebook and render it as an HTML page.

**This is separate because:**

- Notebook has broken cells and hardcoded paths
- Requires model download to execute
- High risk of blocking if combined with the site setup

**Steps:**
1. Move `bertviz`, `ipykernel` to `[project.optional-dependencies] notebooks`
2. Rename `exploring_the _embedding.ipynb` → `exploring_the_embedding.ipynb`
3. Fix hardcoded paths → HuggingFace Hub
4. Fix or remove broken cells ("NOT WORKING", "WORK IN PROGRESS")

## Design Decisions

### Why not plain HTML?
Jekyll gives us markdown authoring, automatic theming, and navigation. The encoding guide has a lot of prose and code blocks — much easier in markdown.

### Why not MkDocs/Jupyter Book?
Overkill. Jekyll is built into GitHub Pages — zero deployment config. Upgrade later if needed.

### Why pre-generate assets instead of building them in CI?
Avoids needing the full Python environment + FluidSynth + model download in CI. The assets change rarely. Commit them directly.

### Why separate the notebook from the site?
The notebook has its own dependency chain (bertviz, ipykernel) and broken cells. Mixing it with the site setup creates a blocker. Ship the site first, add the notebook later.
